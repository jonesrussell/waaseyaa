# WP03 — Sweep + Spec Hygiene

### T010 — Re-run sweep over `packages/*/src/**`

**Purpose**: Confirm at WP03 time that the planning-time sweep is still complete — i.e., no other framework provider (or any code reachable from a kernel boot) constructs a bound `FieldDefinition` without `targetEntityTypeId`.

**Steps**:

1. From repo root, run:
   ```bash
   rg -n "new FieldDefinition\(" packages/*/src/
   ```
2. For each hit, classify:
   - **BIND PATH** (registered via `EntityType::_fieldDefinitions` or via `FieldDefinitionRegistry::register*`): MUST have `targetEntityTypeId: '...'` set.
   - **NON-BIND PATH** (test fixtures, constraint validation, ad-hoc inspection): no requirement.
3. If a BIND PATH hit lacks `targetEntityTypeId`, **stop**, document the hit, and surface to the user before patching — it expands WP03 scope and the PR description must list it.
4. Capture the full output of step 1 in the WP completion notes.

**Validation**: zero defective bind-path call sites remain.

### T011 — Audit `docs/specs/entity-system.md`

**Purpose**: Ensure the field-binding section explains the alpha.171 invariant: every bound `FieldDefinition` must declare `targetEntityTypeId === EntityType::id`.

**Steps**:

1. Open `docs/specs/entity-system.md`. Search for the section that documents field binding (likely under a heading like "Field Definitions", "Binding", or "Core Fields").
2. If the section already states the invariant clearly, **leave it alone**.
3. If the section is missing, vague, or describes a pre-alpha.171 contract, add or replace a single short paragraph:

   > **Field binding invariant (alpha.171+)**: Every `FieldDefinition` registered against an entity type must declare `targetEntityTypeId` equal to that entity type's id. `FieldDefinitionRegistry::registerCoreFields()` and `registerBundleFields()` enforce this at registration time, throwing `\InvalidArgumentException` on mismatch (including the empty-string default).

4. Do **not** rewrite or restructure surrounding sections. The edit should be ≤ 5 lines.

**Validation**: the section reads correctly under both alpha.171 and alpha.172 semantics.

## Risks & Mitigations

- **R-G**: The sweep finds an additional defect not seen at planning time (e.g. a new provider that landed between planning and WP03 execution). *Mitigation*: stop, surface the finding, expand WP03 scope explicitly in the PR description; do **not** silently fold it into a generic "sweep" bullet.
- **R-H**: The drift detector flags an unrelated spec because of incidental file touches in WP02 or WP01. *Mitigation*: investigate the linked spec; if it really is stale, escalate as out-of-scope follow-up issue rather than expanding WP03.
- **R-I**: `entity-system.md` already has the right paragraph but in a different wording than the example above. *Mitigation*: leave it alone — the goal is "the spec correctly states the invariant", not "the spec uses our exact wording".
