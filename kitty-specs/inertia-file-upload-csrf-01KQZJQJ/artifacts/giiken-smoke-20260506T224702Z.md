## Observed behavior

For each multipart POST, the server-side stack trace (captured in the SSR error overlay rendered into the page after submit) shows control reaching `App\Http\Controller\ManagementController::ingestUpload()` at frame `#3`. **For control to reach `ingestUpload()`, the request had to pass `CsrfMiddleware`** — without the WP01+WP02 fix, multipart/form-data without an accepted CSRF token returns 403 at the middleware layer; the controller is never invoked.

The frames at `#4`–`#9` reference paths under `/home/jones/dev/waaseyaa/.worktrees/inertia-file-upload-csrf-01KQZJQJ-lane-b/packages/...`, proving the running server is loading the framework code from the lane-b worktree via the path-repo symlink, not from giiken's published-version vendor directory.

The browser-side `document.cookie` was inspected via Playwright `browser_evaluate` and contains the new `XSRF-TOKEN` cookie:

```
XSRF-TOKEN=ab9699fee87c274ceb33c01d5e4bc68131eeab21102567fe817b852b19b3c3ad
```

This is the smoking gun for FR-001 (token reachable to JavaScript). Inertia's axios reads this cookie name automatically and forwards its (URL-decoded) value as the `X-XSRF-TOKEN` header on every state-changing request. No giiken JavaScript was modified to make this work.

The 500 responses on uploads #2 and #3 came from `MarkItDownConverter::toMarkdown()` (`/home/jones/dev/giiken/src/Ingestion/Converter/MarkItDownConverter.php:38`), which expects a Python binary at `storage/markitdown-venv/bin/markitdown`. That binary is not installed on this dev host (giiken provides `bin/setup-markitdown.sh` to install it). This 500 is downstream of the CSRF gate this mission is fixing and is unrelated to framework correctness.

## Why this proves the mission

**Pre-fix behavior** (verified by the user against `waaseyaa/* ^0.1.0-alpha.173` earlier on 2026-05-06): the same multipart upload from the same Inertia client returned **403 Invalid Security Token** at the CSRF middleware. The mission's spec describes this exact failure mode.

**Post-fix behavior** (this smoke run, against lane-b): the same multipart upload **passes CSRF**, reaches the application controller, and only fails downstream on an unrelated dev-environment dependency.

The fix lives entirely in `waaseyaa/user` and `waaseyaa/foundation` (touched by WP01 and WP02). No giiken code was touched. The "zero consumer-side wiring" guarantee in FR-008 holds: Inertia consumers benefit automatically from the new `XSRF-TOKEN` cookie + `X-XSRF-TOKEN` header convention.
