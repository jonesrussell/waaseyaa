### Why 500 is positive evidence (not negative)

The 500 originated in the **application layer** (`MarkItDownConverter::toMarkdown()`, called from `IngestionHandlerRegistry::handle()` after the controller method `ManagementController->ingestUpload()` ran).

For the request to reach `ingestUpload()`, it MUST have:

1. **Passed `CsrfMiddleware`** — without the WP01+WP02 fix, this would have returned 403 "Invalid Security Token" with the multipart/form-data Content-Type.
2. **Inertia's axios** auto-forwarded the `XSRF-TOKEN` cookie as the `X-XSRF-TOKEN` request header (Inertia + axios convention; no giiken JS code was modified).
3. **The framework code on the lane-b worktree** loaded — the server-log stack trace explicitly references `/home/jones/dev/waaseyaa/.worktrees/inertia-file-upload-csrf-01KQZJQJ-lane-b/packages/...` paths.

The MarkItDown 500 is an unrelated dev-environment issue (`bin/setup-markitdown.sh` not run on this host). It is downstream of the CSRF gate this mission is fixing.
