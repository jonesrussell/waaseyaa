# WP04 — Cross-repo Giiken Ingestion smoke (HARD ACCEPTANCE GATE)

## Objective

Prove the framework change actually reaches the consumer surface. A real Inertia multipart file upload through the giiken Ingestion UI succeeds with **zero giiken-side code changes** (the verification-only path-repo composer config is reverted at the end and leaves no trace).

**This WP is the mission's hard acceptance gate.** The mission cannot be marked done without all four evidence files in `artifacts/`.

## Context

- **Procedure**: Already specified step-by-step in `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/quickstart.md` §5. This WP executes that procedure and captures evidence.
- **Spec**: C-005, NFR-005, SC-4.
- **Giiken state**: clean working tree on `main`, framework version pinned in `composer.json`. The path-repo step temporarily replaces the published version; the revert step restores it.
- **Consumer surface**: The Sagamok Anishnawbek community Ingestion page at `/sagamok-anishnawbek/ingestion` (or current equivalent — verify in the giiken UI on first navigation).

This WP runs after WP01 and WP02 are merged so the framework code on `main` is the version we test.

## Branch Strategy

- Planning base: `main`
- Merge target: `main`
- Lane A, after WP02. The execution worktree path is allocated by `finalize-tasks` and surfaced in `lanes.json`. Use `spec-kitty agent action implement WP04 --agent <name>` to enter the lane.
- All file modifications in this WP are inside `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/artifacts/` (this is `execution_mode: planning_artifact`). The temporary giiken `composer.json` change is **not** owned by this WP — it is reverted before the WP completes.

## Subtasks

### T017 — Wire giiken to local framework (path repository)

**Purpose**: Make giiken use the local waaseyaa monorepo for the smoke run, without committing the change to giiken.

**Steps**:

1. `cd /home/jones/dev/giiken`
2. `cp composer.json composer.json.smoke-backup` (preserve original).
3. Edit `composer.json`. At the **top** of the `repositories` array (or create one if absent), add:

   ```json
   {
     "type": "path",
     "url": "/home/jones/dev/waaseyaa/packages/*",
     "options": { "symlink": true }
   }
   ```

4. `composer update 'waaseyaa/*' --no-progress --no-interaction`.

**Validation**:

- [ ] `composer.json.smoke-backup` exists.
- [ ] `composer update` completes without error.
- [ ] No other giiken files modified.

### T018 — Verify symlink + framework SHA

**Purpose**: Confirm the upload will actually exercise the new framework code.

**Steps**:

1. `ls -la /home/jones/dev/giiken/vendor/waaseyaa/user` — must show a symlink pointing to `/home/jones/dev/waaseyaa/packages/user`.
2. From `/home/jones/dev/waaseyaa`: `git rev-parse HEAD` — capture the SHA. This must include the WP01 commit and WP02 commit.
3. Verify `packages/user/src/Middleware/CsrfMiddleware.php` in the symlinked location contains the `X-XSRF-TOKEN` acceptance code (grep for `X-XSRF-TOKEN`).

**Validation**:

- [ ] `vendor/waaseyaa/user` is a symlink to the local source.
- [ ] Framework HEAD SHA captured for the smoke summary file.
- [ ] `X-XSRF-TOKEN` string is present in the symlinked `CsrfMiddleware.php`.

### T019 — Run giiken; navigate to Ingestion page

**Purpose**: Get to the page that exercises the bug.

**Steps**:

1. `cd /home/jones/dev/giiken`
2. `./vendor/bin/waaseyaa migrate` (idempotent).
3. `./vendor/bin/waaseyaa serve` — runs at `127.0.0.1:8080` (or `php -S 127.0.0.1:8080 -t public public/index.php`).
4. In a browser, open `http://127.0.0.1:8080/`.
5. Log in as the seeded admin/staff user (credentials in giiken `tests/` fixtures or `seeders/`).
6. Navigate to the Sagamok Anishnawbek community management page → Ingestion.
7. Open the browser DevTools Network tab before the upload.

**Validation**:

- [ ] Migrate succeeds (no schema drift).
- [ ] Server reachable at `http://127.0.0.1:8080/`.
- [ ] Login successful.
- [ ] Ingestion page renders.

### T020 — Perform real upload; capture three evidence files

**Purpose**: Capture proof the upload works through the new framework code.

**Steps**:

1. Pick a small test file: `.md` or `.csv`, ≤10 KB. (Example: copy `README.md` from any sub-directory.)
2. Upload it through the Inertia form on the Ingestion page.
3. Capture **three** files (use a single UTC timestamp `<utc>` formatted as `YYYYMMDDTHHMMSSZ` for all three names so they sort together):
   - **Screenshot**: `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/artifacts/giiken-smoke-<utc>.png` — browser screenshot showing the success state after upload.
   - **Network capture**: `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/artifacts/giiken-smoke-<utc>-network.txt` — DevTools right-click → Copy → Copy as cURL on the multipart POST request, paste into the file. The cURL command MUST show the `X-XSRF-TOKEN: ...` header. Add the response status code and headers as a footer.
   - **Server log**: `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/artifacts/giiken-smoke-<utc>-server.log` — excerpt from the server stdout/stderr or log file showing the request handled and a `knowledge_item` row created (grep for the new ID or "INSERT INTO knowledge_item").

**Validation**:

- [ ] Upload returns 200 or 302 (not 403).
- [ ] All three evidence files exist in `artifacts/`, all share the same `<utc>` timestamp.
- [ ] The cURL capture shows `X-XSRF-TOKEN` (not `X-CSRF-Token`) — this is the smoking gun proving the new code path was used.
- [ ] The server log shows a `knowledge_item` row created with the file's content.

**Validation**:

## Risks

| Risk | Mitigation |
|---|---|
| Path-repo symlink doesn't pick up local changes | T018 verifies the symlink resolves AND the new code is actually present in the symlinked file. |
| Forgetting to revert giiken composer.json | T022 explicit `git status` clean check; T023 final checklist re-verifies. |
| Smoke "succeeds" but evidence files missing | T023 explicit existence check on all four files. |
| Token rotation race during smoke (login rotates token) | Smoke flow logs in first, then uploads — token is stable for the upload step. |
| Composer cache uses stale version | If `composer update` doesn't pull local changes, `rm -rf giiken/vendor/composer/installed.json && composer install` to force refresh. |
| Server uses a different framework path than vendor | T018 ensures the symlink is the exact path `vendor/waaseyaa/user`; framework loader uses vendor autoload. |
