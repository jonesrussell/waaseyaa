# WP01 — CSRF middleware: accept X-XSRF-TOKEN + write XSRF-TOKEN cookie

## Objective

Land the framework-side change that closes the gap blocking Inertia file uploads. Two halves:

1. **Request-side**: Extend `CsrfMiddleware` to accept the token from `X-XSRF-TOKEN` (URL-decoded) alongside the existing `_csrf_token` POST field and `X-CSRF-Token` header.
2. **Response-side**: Write an `XSRF-TOKEN` cookie on every HTML response so the token has a JS-reachable surface.

Cover the new behavior with full unit tests pinning the contract in `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/contracts/csrf-token-cookie.md`. Preserve every existing behavior with regression tests.

### T001 — Decide middleware shape (read pass)

**Purpose**: Pick between (a) extend `CsrfMiddleware` to write the cookie on its response path, or (b) introduce a separate `XsrfCookieMiddleware` adjacent to it. Both yield identical observable behavior.

**Steps**:

1. Read `packages/foundation/src/Kernel/HttpKernel.php::handle()` end-to-end. Identify whether middleware are onion-style (`$next($request)` returns a response that the middleware can modify on the way out) or sequential with shared state.
2. Read `packages/user/src/Middleware/CsrfMiddleware.php` end-to-end. Identify whether the existing class can cleanly modify responses on the way out.
3. Decide:
   - If `CsrfMiddleware` already returns a response cleanly → shape (a) (extend existing class).
   - If the kernel uses true onion middleware and adding response-side logic to `CsrfMiddleware` would mix concerns → shape (b) (new `XsrfCookieMiddleware`).
4. Document the chosen shape in the WP commit message body (one paragraph: what you found, why you picked the shape).

**Validation**:

### T002 — Implement cookie writer per contract §1

**Purpose**: Set `Set-Cookie: XSRF-TOKEN=<urlencoded-token>; Path=/; SameSite=Lax[; Secure]` on HTML responses.

**Steps**:

1. Locate the response object exposed by `HttpKernel`. The framework uses `Symfony\Component\HttpFoundation\Response`; cookies are added via `$response->headers->setCookie(Cookie::create(...))`.
2. Build the cookie:

   ```php
   $cookie = Cookie::create('XSRF-TOKEN')
       ->withValue(rawurlencode(CsrfMiddleware::token()))
       ->withPath('/')
       ->withSecure($request->isSecure())
       ->withHttpOnly(false)
       ->withSameSite(Cookie::SAMESITE_LAX);
   ```

   No domain. No `Max-Age`/`Expires` (session cookie).
3. Apply on the response object before it is returned from the middleware/kernel.
4. Honor the existing trusted-proxy logic for `$request->isSecure()` so `X-Forwarded-Proto` is respected when the framework already trusts proxies; do not bypass it.

**Validation**:

- [ ] Cookie value is `rawurlencode($_SESSION['_csrf_token'])`.
- [ ] `Secure` flag tracks `$request->isSecure()` (true on HTTPS, false on HTTP).
- [ ] `HttpOnly=false`.
- [ ] `SameSite=Lax`.
- [ ] `Path=/`.
- [ ] No `Domain` attribute.
- [ ] No `Max-Age` / `Expires` (session cookie).

### T003 — Restrict cookie write to text/html responses; idempotency

**Purpose**: Don't pollute API responses with the cookie; don't double-set.

**Steps**:

1. Detect the response Content-Type primary type: parse `$response->headers->get('Content-Type')`, take the part before `;`, lowercase, compare against `text/html`.
2. Skip the cookie write entirely on non-HTML responses (`application/json`, `application/octet-stream`, downloads, etc.).
3. Idempotency: if the response already carries an `XSRF-TOKEN` cookie (set by an earlier middleware run), don't overwrite it. Use `$response->headers->getCookies()` to check.

**Validation**:

- [ ] `application/json` response → no `Set-Cookie: XSRF-TOKEN` header.
- [ ] `text/html` response → exactly one `Set-Cookie: XSRF-TOKEN` header.
- [ ] If middleware runs twice → still exactly one `Set-Cookie: XSRF-TOKEN`.

### T004 — Accept X-XSRF-TOKEN header (URL-decoded)

**Purpose**: Make the new cookie's value usable on the request side.

**Steps**:

1. In `CsrfMiddleware`'s validation flow, alongside the existing reads of `$_POST['_csrf_token']` and the `X-CSRF-Token` header, add a third read for the `X-XSRF-TOKEN` request header.
2. URL-decode the `X-XSRF-TOKEN` value once with `rawurldecode()` before comparison.
3. Compare each candidate against `$_SESSION['_csrf_token']` via `hash_equals()`. First match wins; if none match, fall through to the existing 403 path.
4. Order is "any-of" — do not prefer one source such that a wrong value in a "preferred" location overrides a correct value in another.

**Validation**:

- [ ] `X-XSRF-TOKEN: <urlencoded current token>` → request authorized.
- [ ] `X-XSRF-TOKEN: <wrong>` AND `_csrf_token` field correct → request authorized (any-of).
- [ ] `X-XSRF-TOKEN: <wrong>` AND nothing else valid → 403.
- [ ] `_csrf_token` field correct AND no headers → request authorized (existing behavior preserved).
- [ ] `X-CSRF-Token: <correct>` AND nothing else → request authorized (existing behavior preserved).
