# WP02 — Integration test through HttpKernel (Inertia multipart)

## Objective

Prove the contract holds when exercised through the framework's full HTTP pipeline. Boot `HttpKernel` end-to-end; perform a GET, observe the cookie; perform a multipart POST with the cookie's URL-decoded value as `X-XSRF-TOKEN`; assert the request is authorized. Add negative and regression cases.

This WP closes NFR-003 (integration coverage).

## Context

- WP01 is the prerequisite. The cookie writer and `X-XSRF-TOKEN` acceptance must already be in place.
- **Pattern to follow**: `tests/Integration/Phase13/SsrHttpKernelIntegrationTest.php`. It already boots `HttpKernel` with a temp filesystem mount and dispatches requests through the full middleware stack. Reuse the setup/teardown pattern; do not invent a new harness.
- **Binding contract**: `kitty-specs/inertia-file-upload-csrf-01KQZJQJ/contracts/csrf-token-cookie.md` §1, §2, §3.

## Subtasks

### T008 — Scaffold InertiaMultipartCsrfIntegrationTest.php

**Purpose**: Set up the kernel-boot harness and a CSRF-protected route fixture.

**File**: `tests/Integration/Phase13/InertiaMultipartCsrfIntegrationTest.php`

**Steps**:

1. Read `tests/Integration/Phase13/SsrHttpKernelIntegrationTest.php` to copy the setup/teardown pattern (temp filesystem, kernel construction, request dispatch).
2. Create the new test class `InertiaMultipartCsrfIntegrationTest extends TestCase` with `#[CoversClass(CsrfMiddleware::class)]`.
3. In `setUp()`: boot the kernel with a minimal route fixture that has at least one CSRF-protected POST endpoint accepting multipart and one accepting JSON.
4. In `tearDown()`: clean the temp filesystem and reset `$_SESSION`.
5. Provide a helper `dispatch(Request $request): Response` that runs the request through `$this->kernel->handle($request)`.

**Validation**:

- [ ] Test class boots without error.
- [ ] Helper dispatches requests through the full middleware stack (no mocked middleware).
- [ ] Teardown resets state between tests.

### T009 — Case 1: GET sets XSRF-TOKEN cookie

**Purpose**: Pin contract §1 through the full pipeline.

**Steps**:

1. Create a `Request` for `GET /test/protected` (an HTML route in the fixture).
2. Dispatch and assert:
   - Response status 200.
   - Response has exactly one `Set-Cookie` header for `XSRF-TOKEN`.
   - Cookie value, when URL-decoded, equals the current `$_SESSION['_csrf_token']`.
   - Cookie attributes match contract §1 (Path=/, SameSite=Lax, HttpOnly absent, Secure tracks scheme, no Domain, no Max-Age).

**Validation**:

- [ ] Test passes.
- [ ] Assertions cover every attribute in contract §1.

### T010 — Case 2: multipart POST with X-XSRF-TOKEN succeeds

**Purpose**: Pin the happy path.

**Steps**:

1. Reuse the kernel state from case 1 (or run case 1's GET first within this test) to obtain the cookie value.
2. URL-decode the cookie value once; this is the value to send as the header.
3. Build a `Request` for `POST /test/protected` with:
   - `Content-Type: multipart/form-data; boundary=...`
   - `X-XSRF-TOKEN: <urldecoded-cookie-value>`
   - A small file upload payload (`file=<bytes>`).
4. Dispatch and assert:
   - Response status 200 or 302.
   - No 403.

**Validation**:

- [ ] Test passes.
- [ ] The token comparison happens server-side via `hash_equals` (proven by sending the wrong-cased equivalent and seeing 403 in case T011).

### T011 — Case 3 (negative): multipart POST without X-XSRF-TOKEN → 403

**Purpose**: Pin the negative half of FR-005.

**Steps**:

1. Build a `Request` for `POST /test/protected` with:
   - `Content-Type: multipart/form-data; boundary=...`
   - **No** `X-XSRF-TOKEN` header.
   - **No** `_csrf_token` field.
2. Dispatch and assert:
   - Response status 403.
   - Response body contains the existing "Invalid Security Token" message (preserve regression).

**Validation**:

- [ ] Test passes.
- [ ] 403 is asserted specifically (not just non-2xx).

### T012 — Case 4 (regression): JSON POST without token still exempt

**Purpose**: Pin FR-004 — `application/json` exemption is unchanged.

**Steps**:

1. Build a `Request` for `POST /test/api/json-route` with:
   - `Content-Type: application/json`
   - **No** CSRF token in any source.
2. Dispatch and assert:
   - Response status is 2xx (route handler runs; CSRF middleware does not block).

**Validation**:

- [ ] Test passes.
- [ ] No regression in JSON exemption.
