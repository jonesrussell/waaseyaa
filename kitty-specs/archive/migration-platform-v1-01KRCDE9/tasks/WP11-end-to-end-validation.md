# WP11 — End-to-end validation: CSV -> entity with resume + rollback

## Objective

Run the entire migration platform end to end: a 1000-record CSV → `migration_test_widget` entities with resume + rollback proven in a single CI-runnable integration test. This is the mission's acceptance gate. If WP11 is green, the substrate is ready for the WordPress reader sibling mission.

The test uses the WP10 `CsvSource` fixture, WP05's `migration_test_widget` entity type fixture, and the full CLI / runner / id-map / run-state / rollback machinery.

## Dependencies

- Internal: WP06 (runner + CLI), WP07 (resume), WP08 (rollback), WP10 (`CsvSource` fixture), WP05 (`migration_test_widget` + `EntityDestination`).
- External: None.
- Charter anchors: validates §5.8 surface as composed.

## Scope (in / out)

**In scope**
- A 1000-record CSV fixture at `packages/migration/tests/Fixtures/data/users-1000.csv` (committed; ~250 KB — acceptable for git).
- A `MigrationDefinition` fixture (`UsersCsvToWidgetsMigration`) declaring the migration: CsvSource → process chain (PassThrough + HtmlSanitize + DefaultValue) → EntityDestination targeting `migration_test_widget`.
- One integration test class with four `#[Test]` methods covering the three FR scenarios (FR-053, FR-054, FR-055) plus an idempotency sanity check.

**Out of scope**
- Multi-migration DAG validation — `import:run-all` is exercised by WP06's unit tests; WP11 stays on a single migration for clarity.
- WordPress source plugin — separate mission.
- Performance benchmarking — out of scope (the run completes in seconds).

## Branch strategy

Planning/base branch: `main`. Merge target: `main`. Per-lane worktree. Run `spec-kitty agent action implement WP11 --agent opus`.

## Implementation guidance

### Subtask T058 — `users-1000.csv` fixture

**Purpose**: A deterministic 1000-row CSV that round-trips through the full pipeline.

**FRs covered**: FR-053 substrate.

**Files**:
- `packages/migration/tests/Fixtures/data/users-1000.csv` (new, ~250 KB committed).

**Steps**:
1. Schema (header row): `id,username,full_name,bio_html,signup_year,tags`. Six columns.
2. 1000 data rows. Deterministic content (seeded random):
   - `id`: integers 1–1000.
   - `username`: e.g. `user-0001` through `user-1000`.
   - `full_name`: random pick from a small dictionary of first+last names (use a 20-entry list, repeated).
   - `bio_html`: random HTML snippet with mixed allowed tags (`<p>`, `<a>`, `<strong>`) and at least 20 rows including a `<script>` tag to verify sanitization.
   - `signup_year`: integers 2010–2025.
   - `tags`: pipe-delimited list of 1–3 tags from a small vocabulary.
3. The committed file can be generated by a small helper script under `packages/migration/tests/Fixtures/data/generate-users-1000.php` (optional — document the seed so others can re-generate identically).

**Validation**:
- [ ] File exists at the documented path.
- [ ] File parses cleanly via `fgetcsv()` (no malformed rows).
- [ ] Row count is exactly 1000.

**Edge cases**:
- Commit the file binary-clean (no BOM, LF line endings).

### Subtask T059 — `UsersCsvToWidgetsMigration` definition fixture

**Purpose**: A realistic `MigrationDefinition` exercising every process plugin and the full destination wiring.

**FRs covered**: FR-053 substrate.

**Files**:
- `packages/migration/tests/Fixtures/Migrations/UsersCsvToWidgetsMigration.php` (new, ~140 lines).

**Steps**:
1. Define a static factory `create(): MigrationDefinition`:
   ```php
   return new MigrationDefinition(
       id: 'users_csv_to_widgets',
       source: new CsvSource($path, keyFields: ['id']),
       process: [
           'title'     => 'username',
           'body'      => new HtmlSanitizeProcessor('bio_html'),
           'value_int' => [new PassThroughProcessor('signup_year'), new TypeCoerceProcessor('int')],
           'tags'      => new DefaultValueProcessor([]),
       ],
       destination: $entityDestinationFactory->forEntityType('migration_test_widget'),
   );
   ```
2. The migration deliberately exercises: PassThrough (string shorthand), HtmlSanitize (with XSS attempts in the fixture), a chain (`['signup_year', TypeCoerce]`), DefaultValue (for `tags` — since CSV has tags, this exercises pass-through; consider including some null rows to exercise the default).

**Validation**:
- [ ] Factory returns a `MigrationDefinition` that passes constructor validation.
- [ ] Discovery via the test's local provider registers the definition correctly.

### Subtask T060 — Integration test class

**Purpose**: The single integration test covering FR-053, FR-054, FR-055.

**FRs covered**: FR-053, FR-054, FR-055.

**Files**:
- `packages/migration/tests/Integration/EndToEndCsvImportTest.php` (new, ~360 lines).

**Steps**:
1. `final class EndToEndCsvImportTest extends \PHPUnit\Framework\TestCase` (`#[CoversNothing]`).
2. `setUp()`:
   - In-memory SQLite DB (`DBALDatabase::createSqlite()`).
   - Run id-map migration (WP04), run-state migration (WP07), `migration_test_widget` schema (WP05 fixture).
   - Register `UsersCsvToWidgetsMigration` via a test-local provider.
   - Construct `MigrationRunner`, `MigrationIdMap`, `MigrationRunState`, `RollbackWalker` with the in-memory DB.
   - Construct a `MigrationLock` pointing at a per-test temp directory (so concurrency tests don't bleed across runs).
3. **Test 1 — `testFullImport`** (FR-053):
   - Run via `MigrationRunner::run('users_csv_to_widgets', RunOptions())`.
   - Assert `RunReport`: `total === 1000`, `imported === 1000`, `failed === 0`, `skipped === 0`.
   - Assert 1000 entities exist in storage (count via `migration_test_widget` storage handler).
   - Assert 1000 id-map rows.
   - Assert 1000 run-state rows all `'success'`.
   - Sanity-check field values on row 42: `title` matches `user-0042`, `body` is sanitized (no `<script>` tag), `value_int` is the parsed signup_year.
4. **Test 2 — `testResumeAfterInterruption`** (FR-054):
   - Run with `RunOptions(limit: 500)`. Assert 500 entities, 500 id-map rows.
   - Capture the `runId` from the partial run.
   - Call `MigrationRunner::runResume('users_csv_to_widgets', RunOptions())`.
   - Assert final state: 1000 entities, 1000 id-map rows, all `'success'` in run-state. Asserts the `runId` is reused across the two batches (FR-037 contract).
5. **Test 3 — `testRollbackClearsEverything`** (FR-055):
   - Run the full import (build on Test 1's setup or re-run inline).
   - Call `RollbackWalker::rollback('users_csv_to_widgets')`.
   - Assert 0 entities, 0 id-map rows, `RollbackReport.rolledBack === 1000`.
   - Assert reverse-creation order: by snapshotting `BeforeDeleteEvent` order, the first delete is the highest-id record (1000), the last delete is record id 1.
   - After rollback, calling `MigrationRunner::runResume(...)` raises `\InvalidArgumentException` ("no prior run found in run-state"). Document — this is correct behavior because rollback does not clear run-state by default; the test should assert against the post-rollback status as visible to operators. Alternatively `import:reset` is the way to wipe run-state cleanly — verify with `MigrationRunState::countByStatus()` whether run-state survived rollback (per WP08 design, it should survive — operators can audit a rolled-back migration).
6. **Test 4 — `testIdempotentReRun`** (sanity beyond FR-053..FR-055):
   - Run twice in a row (no rollback between). Assert: 1000 entities, 1000 id-map rows (no duplicates), second run's `RunReport.imported === 0` and `.skipped === 1000` (FR-030, FR-031).

**Validation**:
- [ ] All four tests green.
- [ ] Full suite green.

**Edge cases**:
- Test 2's setUp must commit the run-state rows from the first partial run BEFORE the resume call. Verify by reading the run-state table between the two calls.
- Test 3's reverse-order snapshot must avoid bleeding event subscribers across tests — register per-test.
- `EntityDestination` requires an `AccountInterface` that passes the `create` gate — the setUp wires a `DevAdminAccount` for this purpose (or equivalent).

## Risks

- **R1 — Slow test on cold cache**: in-memory SQLite + 1000 row inserts × 1000 events × 1000 id-map upserts may take noticeable wall-time. Budget 60 seconds. If it exceeds, document and revisit batch-mode (deferred to a future flag).
- **R2 — Memory leak in long-running test**: 1000 records iterated lazily should stay under 50 MB. Assert via `memory_get_peak_usage(true)` in `tearDown()`.
- **R3 — Event subscriber bleed**: each test must reset subscribers in `setUp()`. Otherwise Test 3's reverse-order snapshot is contaminated by Test 1's events.
- **R4 — `--limit` semantics ambiguity on resume**: Test 2 uses `--limit=500` for the partial run; the resume call uses no `--limit`. Verify that the resume's limit isn't accidentally inherited from the prior run's `RunOptions`.
- **R5 — Run-state survives rollback**: per WP08 design choice (operators may want to audit rolled-back migrations). Test 3 verifies this is intentional; document in WP12 the operator-friendly recovery path (`import:reset` cleans run-state).
