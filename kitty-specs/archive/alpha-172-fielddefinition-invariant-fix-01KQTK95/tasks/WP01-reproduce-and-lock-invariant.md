# WP01 — Reproduce + Lock Invariant

## Objective

Land four failing tests that reproduce GitHub issue #1388 and lock the alpha.171 `FieldDefinition` binding invariant against future regressions. After this WP merges (still red), WP02 will turn the suite green by patching the three defect sites.

This is the foundation WP. It encodes the bug as a test, encodes the invariant contract as a test, and gives every downstream WP its acceptance signal.

## Context

- **Spec**: [`spec.md`](../spec.md) — see FR-004, FR-005, FR-006, NFR-002.
- **Plan**: [`plan.md`](../plan.md) — Phase 1 § "Test design".
- **Research**: [`research.md`](../research.md) — R2 (exception contract), R3 (defect sites).
- **Issue**: [waaseyaa#1388](https://github.com/jonesrussell/waaseyaa/issues/1388).
- **Bind-time invariant**: `Waaseyaa\Field\FieldDefinitionRegistry::registerCoreFields()` (line 39) and `registerBundleFields()` (line 127) throw `\InvalidArgumentException` when `$field->getTargetEntityTypeId() !== $entityTypeId`.
- **Exception message format** (paraphrased): `Core field "<name>" declares targetEntityTypeId "<actual>" but is being registered against entity type "<expected>".`
- **Canonical clean call site to mirror**: `packages/genealogy/src/GenealogyFieldDefinitions.php`.

## Subtasks

### T001 — Provider unit test for `GroupsServiceProvider`

**Purpose**: Codify the expectation that `GroupsServiceProvider::register()` registers `group_type` cleanly and that every core field declared on it carries `targetEntityTypeId === 'group_type'`.

**File**: `packages/groups/tests/Unit/GroupsServiceProviderTest.php` (NEW). Place under `Waaseyaa\Groups\Tests\Unit` namespace.

**Steps**:

1. Construct a minimal kernel-like wiring that lets `GroupsServiceProvider::register()` run. Look at existing tests in `packages/groups/tests/` (or, if absent, mirror the closest analogue in `packages/taxonomy/tests/` or `packages/genealogy/tests/`) for the canonical setup.
2. Call `register()` on the provider.
3. Assert `EntityTypeManager::getDefinition('group_type')` returns a non-null `EntityType` (the registration didn't throw).
4. Iterate the entity type's `_fieldDefinitions`. For each `FieldDefinition`:
   - assert `$field->getTargetEntityTypeId() === 'group_type'`
   - in particular, name a parameterized data set that includes the `description` field as the regression case from #1388.
5. Use PHPUnit attributes: `#[Test]`, `#[CoversClass(\Waaseyaa\Groups\GroupsServiceProvider::class)]`.

**Validation**:
- Pre-WP02: this test fails on step 3 (or step 4 for `description`) with the documented `\InvalidArgumentException`.
- Post-WP02: this test passes on every assertion.

### T002 — Provider unit test for `TaxonomyServiceProvider`

**Purpose**: Same as T001, for `TaxonomyServiceProvider` / `taxonomy_vocabulary`. Must catch BOTH defective fields (`description` and `weight`).

**File**: `packages/taxonomy/tests/Unit/TaxonomyServiceProviderTest.php` (NEW). Namespace `Waaseyaa\Taxonomy\Tests\Unit`.

**Steps**:

1. Mirror the structure of T001.
2. Iterate every core `FieldDefinition` on `taxonomy_vocabulary` and assert `getTargetEntityTypeId() === 'taxonomy_vocabulary'`.

**Validation**: same shape as T001.

### T003 — Registry-level negative test

**Purpose**: Pin the `\InvalidArgumentException` contract that `FieldDefinitionRegistry` throws on invariant violation. The manifest sweep (T004) and the provider tests (T001/T002) all rely on this exception class and message format being stable.

**File**: `packages/field/tests/Unit/FieldDefinitionRegistryInvariantTest.php` (NEW). Namespace `Waaseyaa\Field\Tests\Unit`.

**Steps**:

1. Construct a `FieldDefinition` with `targetEntityTypeId: ''` (empty default).
2. Build a `FieldDefinitionRegistry` (whatever its constructor requires — read the source to confirm).
3. Use `$this->expectException(\InvalidArgumentException::class)` and `$this->expectExceptionMessageMatches('/declares targetEntityTypeId "" but is being registered against entity type "/')`.
4. Attempt to register the field as a core field on a non-empty entity type id (e.g. `'sample_entity'`).
5. Add a second test method covering the bundle-fields path (`registerBundleFields()` at line 127) using the same exception expectations. The bundle path message uses the `FieldDefinition` prefix instead of `Core field`; assert against the right substring.

**Validation**: passes today (the registry already throws); locks the contract going forward.

### T004 — Manifest-level integration sweep

**Purpose**: Boot a minimal kernel and assert that every entity type registered in `EntityTypeManager` has every bound `FieldDefinition` declaring a non-empty `targetEntityTypeId` equal to the entity type's id.

**File**: `tests/Integration/PhaseN/FieldDefinitionInvariantTest.php` (NEW). Namespace `Waaseyaa\Tests\Integration\PhaseN`. Use whatever phase number is conventional for new mission tests in `tests/Integration/` — pick the next free `PhaseN` directory or place under an existing one if conventions allow.

**Steps**:

1. Boot a kernel context (re-use the smallest existing harness in `tests/Integration/`; don't write new bootstrap code unless required).
2. Register the framework providers under test:
   - `waaseyaa/foundation`
   - `waaseyaa/entity`
   - `waaseyaa/field`
   - `waaseyaa/groups`
   - `waaseyaa/taxonomy`
   Add others only if the kernel actually requires them to boot the manifest path.
3. After registration, walk every entry in `EntityTypeManager::getDefinitions()`.
4. For each entity type: walk its `_fieldDefinitions` and assert:

   - `$field->getTargetEntityTypeId() === $entityType->id()` — same failure message convention.
5. Use `#[CoversNothing]` per the integration-test convention in this repo.
6. Keep the test under 5 seconds wall-clock (NFR-002).

**Validation**: pre-WP02 this fails on first hit (Groups or Taxonomy depending on registration order); post-WP02 it passes for every entity type in the framework manifest.
