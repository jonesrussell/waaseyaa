# WP08 Review Feedback — Reject

**Verdict**: Reject (one defect; otherwise solid).

## Defect: Header row is emitted as a data row

`GfmTableParser::parse()` (`packages/structured-import/src/Gfm/GfmTableParser.php:101-102`) appends the header cells to `$rows` as the first output entry:

```php
// The header row is the first output row.
$foundTable = true;
$rows[] = ['prompt' => $headerCells[0], 'value' => $headerCells[1]];
```

The class-level docblock documents this intent: "Row output includes the header row as the first item." The 18 unit tests assert the same behavior. For example, `tests/Unit/Gfm/GfmTableParserTest.php:32-39`:

```php
$payload = "| Title | Hello |\n| --- | --- |\n| Body | Lorem |";
// expected:
[
    ['prompt' => 'Title', 'value' => 'Hello'],
    ['prompt' => 'Body',  'value' => 'Lorem'],
]
```

In a GFM table, the row above the `:?-+:?` separator is the **column-name row**, not user data. In the F5 import context the header is "Field" / "Value" (or "Prompt" / "Value", or — as in some tests — "Title" / "Hello" used as illustrative column labels). Feeding "Field" or "Title" into prompt-alias matching in WP09 produces a false unmatched row at best, and a false positive match against an actual `title` field at worst. This propagates a real bug into the next work package.

The intended behavior, per `research.md` Q7 and `data-model.md` § 6, is:

```
| Field | Value |     ← column labels (skip)
| ---   | ---   |     ← separator (skip)
| Title | Hello |     ← data row #1
| Body  | Lorem |     ← data row #2
```

So the parser output for the example above must be:

```php
[
    ['prompt' => 'Title', 'value' => 'Hello'],
    ['prompt' => 'Body',  'value' => 'Lorem'],
]
```

…and for `| Field | Content |\n| --- | --- |\n| Title | My Title |\n| Body | My Body |` (test at line 337) it must be:

```php
[
    ['prompt' => 'Title', 'value' => 'My Title'],
    ['prompt' => 'Body',  'value' => 'My Body'],
]
```

## Required fix

1. In `GfmTableParser::parse()`:
   - Do **not** append `$headerCells` to `$rows`.
   - Keep the column-count validation against the header (it still serves to reject non-2-column tables).
   - Optionally: capture the header labels into a separate field on `GfmParseResult` if WP09 wants them for diagnostics, but do not emit them in `rows`.
2. Update the docblock on `GfmTableParser` (lines 18-19) — remove "Row output includes the header row as the first item."
3. Update **every** test in `tests/Unit/Gfm/GfmTableParserTest.php` whose expected `rows` array currently begins with the header row. The affected tests (by line number of expected array) include at least:
   - L37-38 (happy path with pipes)
   - L53-54 (happy path without pipes)
   - L70-71, L86-87 (alignment markers)
   - L106-107 (escaped pipe)
   - L123-124 (escaped pipe in header)
   - L145-146 (3-col data row in 2-col table — keep the error, drop the header from expected rows)
   - L213-214, L279-280, L296-297, L317-318, L349-351 (multi-table / prose-surrounded fixtures)

   In each case, drop the first element (the header) from the expected `rows` array. The error-list assertions stay as-is.
4. Add one new test that explicitly asserts the header is **not** emitted: e.g. for `| Field | Value |\n| --- | --- |\n| Title | Hello |`, expect `rows === [['prompt' => 'Title', 'value' => 'Hello']]`.
